# 《金铲铲之战》怪兽入侵 17.6b 机制知识审计

研究日期：2026-07-15  
目标版本：金铲铲移动端，返场“怪兽入侵”，标准排位/标准匹配，17.6b  
项目数据版本：`17.17.6b-S18`，`mode=8`  
研究目的：在继续比赛顾问、强化符文刷新建议和实时 LLM 对话之前，建立一套可追溯、可版本化、不会把旧版或云顶规则误当成当前金铲铲规则的机制基线。

## 1. 结论摘要

当前项目拥有棋子、羁绊、装备、强化符文和阵容数据，但还没有完整的“对局规则层”。这会导致算法知道“有什么”，却不知道“它为什么会在此时出现、还能不能刷新、选择后会改变什么”。

已经确认的高影响问题：

1. 当前返场版不是简单复刻 2022—2023 年 S8/S8.5。17.5b 新增城市守护者、开场奇遇、棱彩羁绊与三星五费升级，并恢复旧英雄定位、9 级商店上限和受击回复法力值。
2. 城市守护者在标准模式中是正式经营变量：开局获得守护者蛋，2-7 从 3 个随机守护者中选 1 个；本局所有玩家刷到的守护者形态一致，只会全是战力形态或全是经济形态。
3. 当前项目完全没有记录城市守护者、开场奇遇、强化提案和重随资源，因此无法正确判断金币、经验、装备、商店概率和阵容强度。
4. 当前项目的商店基础数据已经出现可证伪错误：`public/odds-data.js` 声称五费棋子有 9 种，而 17.6b 官方棋子数据实际只有 8 个不同五费棋子。
5. `poolCopies` 只是沿用旧 S8 份数，当前没有 17.6b 证据。涉及 D 牌成本和“刷新是否值得”的结论都必须降级为未验证。
6. 2026 返场客户端已经直接确认：普通强化三个栏位各有 1 次独立重随；英雄强化三个栏位各有 3 次独立重随。原版怪兽入侵“4 次共享重随”没有继承，必须标记为旧规则，不能再作为当前建议依据。
7. 项目商店概率表不是当前返场表。当前客户端可确认基础 8 级为 `16/20/35/25/4`，基础 9 级为 `9/15/30/30/16`；`5/10/20/40/25` 是携带【高端购物】后 9 级套用的虚拟 10 级概率，不是基础 9 级概率。
8. 当前比赛页仍然只能比较“三个现有强化中哪个最好”，不能基于已见选项、剩余分栏重随次数和后续候选分布比较“拿还是刷”。在补齐状态机与候选模型前，不应直接输出“建议刷新”。

## 2. 研究边界

### 2.1 本规则集包含

- 返场怪兽入侵的标准排位与标准匹配。
- 当前 17.6b 棋子、羁绊、装备、强化、开场奇遇和城市守护者。
- 对比赛顾问有决策意义的回合、商店、经济、强化和特殊机制。

### 2.2 必须隔离的模式

- 双人模式：包含魄罗传送器、高星弈子传送和队友支援规则。
- 怪兽入侵·恭喜发财：包含 1-2 高费开局、胖胖龙宝库等独立机制。
- 狂暴模式及其他娱乐模式。

任何来自这些模式的数据都不能混入标准模式规则集。

### 2.3 证据等级

| 等级 | 定义 | 可否直接进入规则引擎 |
|---|---|---|
| A | 当前版本腾讯官方游戏数据或官方公告正文 | 可以 |
| B | 完整转载官方更新公告、且多来源文本一致 | 可以，但保留转载标记 |
| C | 腾讯/Riot 旧版官方规则 | 只可作为历史继承候选 |
| D | 攻略、视频、社区观察 | 口播结论只能交叉验证；当前客户端 UI 中直接可读的数字可确认“画面可见状态”，但不能外推隐藏规则 |

机制状态统一使用：

- `confirmed`：当前返场版已确认。
- `version-dependent`：旧版或通用规则明确，但返场版是否继承仍需确认。
- `unverified`：项目或社区存在说法，缺少足够证据。
- `conflict`：现有数据之间互相矛盾。
- `deprecated`：旧规则或其他模式规则，不适用于当前标准模式。

## 4. 城市守护者

### 4.1 基础规则

当前玩法总览图可以确认：

1. 开局固定获得一枚守护者蛋，放入备战席。
2. 在 2-7 回合，从 3 个随机城市守护者中选择 1 个。
3. 城市守护者不占人口，拥有特殊功能。
4. 分为战力形态和经济形态。
5. 同一局所有玩家出现的形态一致：要么全战力，要么全经济。

这意味着比赛状态至少需要：

```text
guardianOffer.form
guardianOffer.options
guardian.selected
guardian.progress
guardian.cooldowns
guardian.pendingRewards
```

## 6. 强化符文与英雄强化

### 6.1 当前数据能确认什么

`data/hex.js` 当前有 322 条强化：

- 白银阶：39
- 黄金阶：91
- 棱彩阶：70
- 英雄强化：122

当前字段只有：

```text
id, name, desc, icon, level, hero_enhancement_type, fetterId, fetterType
```

缺少：

- 可出现回合。
- 版本启用/禁用区间。
- 普通强化或英雄强化的正式类型字段。
- C 位/团队型的可靠分类。
- 互斥组。
- 当前提案、已见选项、重随资源。
- 重随后的排除与量身定做规则。

### 6.2 普通强化重随

2023 年腾讯官方《符文大陆传说赛季机制概览》明确：

> 每个强化符文都能单独重随，不再重随整组三个选择。

该中文页面还写了“每场对局总共有18次强化重随”，但这与三个阶段、三个栏位的数量关系存在歧义；英文原意很可能是在说每局总共可看到 18 个强化选项，而不是 18 次按钮操作。

2026-07-05 的返场标准模式客户端画面进一步直接确认：

- 2-1 普通强化首次打开时，三个栏位下方均显示 `1` 次重随。
- 重随其中一个栏位后，该栏位变为 `0`，另外两个栏位仍为 `1`。
- 3-2 再次打开普通强化时，三个栏位重新显示为 `1`。

因此当前规则可以写死为：普通强化每个提案阶段、每个栏位各 1 次独立重随，并在下一次强化提案时重置。

### 6.3 原版怪兽入侵英雄强化重随

13.3 官方公告明确：

- 英雄强化出现时，有 4 次英雄强化重随机会。
- 不消耗普通强化的重随机会。
- 全部用完可看到 15 个不同英雄强化。
- 重随不会重复已经见过的英雄强化。

这是原版怪兽入侵的确定规则，但 2026 返场客户端已经证明没有继承这套资源模型：

- 2026-06-17 的 2-1 英雄强化页面同时出现 `3 / 2 / 0` 三个栏位计数，总剩余次数大于 4，排除了“全页面共享 4 次”的可能。
- 2026-07-13 的英雄强化首次提案画面中，三个栏位均显示 `3`。
- 2026-07-08 的英雄强化页面也能看到单个英雄强化栏位独立显示剩余次数。

当前返场规则应改为：英雄强化每个栏位各有 3 次独立重随。旧版“共享 4 次、最多看 15 个不同英雄强化”应标记为 `deprecated`。至于重随后是否仍保证不出现已见选项，现有画面只证明短序列内没有立即重复，仍不足以确认完整排除规则。

### 6.4 原版英雄强化量身定做

13.5 官方公告明确：

- 2-1 英雄强化全随机。
- 3-2 和 4-2 会根据已激活羁绊量身定做。
- 0/1/2/3+ 个已激活羁绊，对应 0/1/2/3 个量身定做栏位。
- 量身定做栏位会随每次重随随机变化。
- 3-2、4-2 不再出现三张完全相同费用的组合。

13.6 又限制了量身定做的怪兽英雄强化数量。

当前 `matcher-core.js` 只写了费用可行域：

```js
2-1: [1, 2, 3]
3-2: [2, 3, 4]
4-2: [3, 4, 5]
```

这不足以表示量身定做、非重复、特殊强化限定回合以及当前已激活羁绊的影响。

### 6.5 正式提案状态

建议状态模型：

```json
{
  "offerId": "aug-3-2-001",
  "round": "3-2",
  "rarity": "gold",
  "kind": "hero",
  "options": [
    {"slot": 0, "augmentId": "...", "seen": true, "rerollsRemaining": 3},
    {"slot": 1, "augmentId": "...", "seen": true, "rerollsRemaining": 3},
    {"slot": 2, "augmentId": "...", "seen": true, "rerollsRemaining": 3}
  ],
  "status": "offered"
}
```

事件必须区分：

```text
augment.offer
augment.reroll
augment.reroll.result
augment.select
```

只有 `augment.select` 才能把强化加入正式阵容信号池。

## 7. 商店、等级与牌池

### 7.1 已确认

- 标准模式商店等级上限为 9。
- 17.6b 官方棋子数据中，不同棋子数量为：
  - 一费 13
  - 二费 13
  - 三费 14
  - 四费 13
  - 五费 8

### 7.2 项目当前冲突

`public/odds-data.js` 当前写的是：

```js
unitCounts: { 1: 13, 2: 13, 3: 14, 4: 13, 5: 9 }
```

五费数量与当前官方数据不一致，应改为证据生成值，不能手工维护。

错误不只存在于展示文件。`src/pipeline/discover.js` 的 `SHOP_ODDS` 和 `public/discovery-data.js` 的生成结果也仍使用旧表，并继续向以下模块传播：

```text
src/pipeline/route-certifier.js
src/pipeline/meta-solver.js
src/pipeline/transition.js
public/index.html
```

因此当前路线认证中的“多少金币搜到二星/三星”“某阶段命中概率”“路线失败率”都会受到系统性偏差；这不是单纯 UI 文案问题。

该文件还写着：

```js
poolCopies: { 1: 29, 2: 22, 3: 18, 4: 12, 5: 10 }
```

注释已经承认这是“S8 同口径”假设。17.6b 公告还修复过爆裂球果导致牌库棋子数量异常的问题，说明返场版确实存在共享牌库，但具体每张份数仍需当前版本证据。

### 7.3 当前客户端概率证据

项目当前商店概率：

```text
3级 75/25/0/0/0
4级 55/30/15/0/0
5级 45/33/20/2/0
6级 25/40/30/5/0
7级 19/30/35/15/1
8级 18/25/32/22/3
9级 10/20/25/35/10
```

其中 8、9 级已被当前客户端直接证伪。

当前可确认的基础概率如下：

```text
3级 75/25/0/0/0
4级 55/30/15/0/0
5级 45/33/20/2/0
6级 25/40/30/5/0
8级 16/20/35/25/4
9级 9/15/30/30/16
```

7 级仍缺少足够清晰的当前客户端画面，不能仅沿用项目中的 `19/30/35/15/1`。

此前观察到的：

```text
9级 5/10/20/40/25
```

不是基础 9 级概率。两段相关画面都带有【高端购物】；当前腾讯 `data/hex.js` 对其描述为“商店会以比当前等级高 1 级的几率刷新英雄”。因此：

- 8 级携带【高端购物】显示 `9/15/30/30/16`，反推出基础 9 级概率。
- 9 级携带【高端购物】显示 `5/10/20/40/25`，这是虚拟 10 级概率。

概率引擎必须把“玩家等级”和“商店有效概率等级”拆开，至少增加：

```text
playerLevel
shopOddsLevel
shopOddsModifiers
```

否则拿到【高端购物】、防御塔经济形态或其他商店修正后，算法会系统性算错搜牌期望。

## 8. 经济与回合时序

当前返场客户端已经确认的升级经验阈值：

```text
4升5：10
5升6：20
6升7：36
7升8：56
8升9：80
```

2 升 3、3 升 4 仍需补当前画面。上述视频使用了【恶魔契约】，它只改变购买经验的支付方式，因此可以读取升级阈值，但不能据此确认普通情况下购买经验的金币/经验比。

以下机制对经营建议仍然至关重要，但本轮没有拿到 17.6b 专门公告，暂不写死：

- 刷新商店金币成本。
- 购买经验的金币/经验比。
- 2 升 3、3 升 4 所需经验。
- 每回合基础收入。
- 利息上限和利息档位。
- 连胜/连败奖励档位。
- 各阶段玩家伤害。
- 选秀与 PvE 的准确回合表。
- 装备掉落保底和补偿规则。

这些规则通常跨赛季稳定，但项目目标是比赛顾问，不能用“通常如此”代替当前版本证据。下一轮研究应优先补齐它们。

## 9. 羁绊与棋子内的可变机制

当前比赛状态不仅要记录羁绊数量，还要记录羁绊内部选择和永久进度：

| 机制 | 必需状态 |
|---|---|
| 战斗机甲 | 至尊机甲是谁、被吸收的两名棋子、当前是否完成选择 |
| AI程序 | 本局条件、已选结果、4/6 层额外结果 |
| 气象主播 | 本局天气类型和强化格位置 |
| 枪神 | 厄斐琉斯当前武器选择 |
| 地下魔盗团 | 当前锁数、奖励层级、收菜或继续 |
| 幻灵战队 | 永久名声层数 |
| 小天才 | 当前临时装备及下回合解体时间 |
| 福牛守护者 | 每名棋子的单场保命是否已触发 |
| 星之守护者 | 额外法力获取倍率 |
| 城市守护者 | 独立进度、奖励和延迟回合 |

仅从“当前激活了 5 战斗机甲”无法恢复这些状态。

## 10. 装备系统

17.5b 公告确认：

- 返场 13 件经典成装。
- 同时加入对应光明装备。
- 当前官方资料中共有 149 条装备数据，包含基础、成装、光明、神器、纹章和特殊消耗品。

比赛顾问需要区分：

- 基础装备、普通成装、光明装备、神器。
- 可合成纹章与不可合成纹章。
- 临时装备和永久装备。
- 锻造器、重铸器、拆卸器、复制器等消耗品。
- 装备商人和小天才产生的临时装备。
- 已承诺给路线但尚未到手的延迟装备。

当前 `selectedItems` 的扁平数组不足以表达这些差异。

## 11. 当前比赛模式的机制缺口

### P0：没有这些状态就不能给可靠建议

- 当前补丁与模式。
- 当前回合。
- 开场奇遇。
- 城市守护者形态、选择、进度和待领取奖励。
- 当前金币、经验、等级与生命值。
- 强化提案、已见选项、每栏剩余重随资源。
- 商店概率的证据状态。
- 牌池份数及外部持有量是否已知。

### P1：会显著改变路线经营

- 连胜/连败状态。
- 地下魔盗团进度。
- AI程序配置。
- 气象主播天气。
- 至尊机甲选择。
- 枪神武器。
- 装备类型、临时性和延迟获取回合。

### P2：战斗解释与精细化建议

- 受击回蓝、索敌、控制免疫、加时规则。
- 玩家伤害和淘汰顺序。
- PvE 战利品分配细节。
- 棱彩羁绊和三星五费的完整特殊行为。

## 12. 研究与实现顺序

1. 冻结当前“建议刷新”功能，不让 LLM 猜次数。
2. 建立版本化规则集和证据状态，本报告对应 `monster-invasion-17.6b`。
3. 让启动新局时先选择模式，并记录开场奇遇。
4. 加入 2-7 城市守护者提案和正式选择事件。
5. 加入强化提案、重随、结果和选择事件。
6. 将普通强化“每栏 1 次”和英雄强化“每栏 3 次”写入提案状态机，并移除共享英雄重随字段。
7. 补齐当前 7 级商店概率、低等级经验、经济表和牌池份数。
8. 完成规则回归场景后，再实现“拿还是刷新”的期望值比较。
9. 最后再让 LLM 用自然语言解释确定性引擎的建议。

## 14. 尚未解决的关键问题

这些问题必须保留为未知，不能由 LLM 补全：

1. 返场英雄强化重随后是否继续保证不重复？
2. 返场是否完整继承 13.5/13.6 的量身定做规则？
3. 当前每费棋子的牌池份数是多少？
4. 当前 7 级基础商店概率是多少？
5. 当前 2 升 3、3 升 4 经验阈值、普通购买经验比、基础收入、利息、连胜/连败和玩家伤害表是什么？
6. 防御塔经济形态提高 4/5 费概率时，具体概率修正和与【高端购物】的叠加顺序是什么？
7. 爆裂球果战力、奶酪首领经济、卑鄙之喉经济的完整当前文本是什么？

现在系统可以可靠读取“普通强化每栏 1 次、英雄强化每栏 3 次”，但在候选排除、量身定做和剩余池分布补齐前，仍不能只凭剩余次数就断言“应该刷”。
